# Отслеживание задержки курсов ЦБ — инструкция для Claude

> Этот файл — для меня (Claude) на будущие сессии. Цель пользователя: курсы ЦБ
> в нашем сервисе выходят позже, чем в платном расширении **stocksi Ultimate**
> (на 30–60 с). Надо догнать или обогнать. Пользователь НЕ хочет отслеживать
> вручную — измерение и разбор делаю я сам по логам.

## TL;DR при старте сессии

1. Открыть `funding-service/logs/cbrwatch-summary.md` — там последний разбор
   гонки каналов ЦБ за каждый день публикации.
2. Если за свежую дату есть строка с публикацией — посмотреть, какой канал был
   первым, наш лаг и ошибку прогноза. Сделать выводы / подкрутить боевой источник.
3. Если логов нет за ожидаемый день — проверить, жива ли Scheduled Task `cbrwatch`
   (см. «Эксплуатация» ниже).

## Почему именно так измеряем

- stocksi Ultimate — закрытое браузерное расширение, публичного API нет, пуллить
  его нельзя. Поэтому stocksi напрямую не наблюдаем.
- Но официальный курс никто не может получить раньше, чем он появится на **каком-то
  из каналов ЦБ** (если только не предсказывать его сами — это наш «прогноз»).
- Значит, достаточно мерить: (1) гонку каналов ЦБ между собой, (2) лаг нашего
  сервиса относительно самого быстрого канала, (3) точность нашего прогноза.
  Самый быстрый канал = потолок для всех, включая stocksi.

## Каналы ЦБ (origin → их и меряем как «эталон скорости»)

| ID | URL | Примечание |
|----|-----|-----------|
| `cbr_official_xml` | `cbr.ru/scripts/XML_daily.asp` | то, что сейчас в бою; кэшируемый статик, обновляется позже всех |
| `cbr_soap` | `cbr.ru/DailyInfoWebServ/DailyInfo.asmx` | читает из БД ЦБ, обычно лидирует |
| `cbr_json_mirror` | `cbr-xml-daily.ru/daily_json.js` | зеркало, может только ОТСТАВАТЬ — в бой не берём |
| `cbr_xml_mirror` | `cbr-xml-daily.ru/daily.xml` | зеркало, то же |

Код фетчеров: `internal/source/cbr/channels.go` (единый источник правды; его же
использует диагностический `GET /api/v1/cbr-race` и боевой `source.go`).

## Что и куда логирует cbrwatch

Бинарь `cmd/cbrwatch`. Опрос: 1 с в окне **16:00–19:00 МСК**, иначе раз в 5 мин.

- `funding-service/logs/cbrwatch-YYYY-MM-DD.jsonl` — сырые события (JSONL), по строке
  на каждое «канал X впервые показал дату D» с временем UTC+МСК. Машиночитаемо — парсю я.
- `funding-service/logs/cbrwatch-summary.md` — человекочитаемый итог по дням: кто
  первый, разброс между каналами (с), наш лаг, прогноз vs факт. Это читаю первым делом.

Опционально cbrwatch снимает снапшот нашего сервиса через `GET /api/v1/snapshot`
(если бэкенд запущен) — чтобы поймать момент, когда НАШ сервис увидел курс, и
зафиксировать `predicted_cb_rate` vs факт.

## Эксплуатация (Scheduled Task)

- Задача Windows: имя `cbrwatch`, триггер «при входе в систему», авто-рестарт.
  Бинарь спит вне окна, нагрузки нет.
- Проверить статус: `Get-ScheduledTask -TaskName cbrwatch`
- Перезапустить: `Start-ScheduledTask -TaskName cbrwatch`
- Пересобрать бинарь после правок:
  `go build -o ../logs/bin/cbrwatch.exe ./cmd/cbrwatch` (из `backend/`)
- Регистрация задачи: скрипт `funding-service/scripts/install-cbrwatch-task.ps1`.

## Как читать итог и что делать

- **SOAP первым, XML_daily на 30–60 с позже** → подтверждает гипотезу: надо в боевом
  `source.go` опрашивать каналы параллельно и эмитить по первому (Part A). Если уже
  сделано — наш лаг должен схлопнуться до ~1 с от лидера.
- **Прогноз близок к факту (ошибка < ~10–20 коп.)** → можно отдавать `predicted_cb_rate`
  ещё ДО публикации ЦБ и обогнать stocksi в принципе. Копим статистику по дням.
- **Все каналы приходят почти одновременно, а stocksi всё равно раньше** → значит они
  предсказывают; упор на наш прогноз (Part C).

### Что ещё можно сделать (не критично)
- Отдельная таблица в БД «прогноз↔факт по дням» для долгой статистики. Сейчас
  это покрывает `cbrwatch-*.jsonl` (event `prediction`) + персист снапшотов в `snapshots`.
- Реальный разброс каналов появится в `summary.md` только ПОСЛЕ живой публикации
  (≈16:30 МСК буднего дня): тогда XML_daily отстанет от SOAP — это и подтвердит фикс.

## Как (пере)запустить вручную
- Разовый замер сейчас: `go run ./cmd/cbrwatch -once` (из `backend/`).
- Перезапустить демон: `Get-Process cbrwatch | Stop-Process; wscript "$env:APPDATA\Microsoft\Windows\Start Menu\Programs\Startup\cbrwatch.vbs"`
- Пересобрать после правок: повторно запустить `install-cbrwatch-task.ps1`.
